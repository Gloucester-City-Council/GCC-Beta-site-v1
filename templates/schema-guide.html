How to build a council service from a content schema

A step-by-step guide for content designers and developers: how to pick the right template, read a content schema, populate pages from authorised data only, and annotate each section so content can be traced back to its source.

Before you start

Before you can build any service page you need two things:

  1. A content schema — a JSON file that captures all the authorised content for the service. This is your single source of truth. Every sentence on your pages must trace back to a specific path in the schema. If it is not in the schema, it must not appear on the page.
  2. An HTML template — a blank, accessible page scaffold from the templates/ folder. Copy the right one for the type of page you are building.

Why use a schema? A content schema separates the what (authorised information, policy text, contact details) from the how (the HTML layout). It makes it straightforward for a reviewer to check that a page contains only approved content — they can compare the schema against the page section by section without reading HTML.

Where to find the templates

Templates live in the templates/ folder at the root of this repository. The folder contains:

  • service-landing.html — Template 1. Hero, service card grid, popular tasks strip.
  • transaction.html — Template 2. "Start now" call to action or inline form.
  • information.html — Template 3. Two-column layout with on-page navigation and sidebar.
  • faq.html — Template 4. Accordion of questions grouped by category.
  • assets/ — Shared CSS and JavaScript. Link to these from every page using a relative path.

Open templates/index.html in a browser to preview all four templates side by side before choosing one.

Choose a template

Each template serves a specific purpose. Pick the one that matches what the user is trying to do on that page.

Template selection guide
Template Use it when… HMO example
Service landing You are introducing a whole service area or a user-type hub that fans out to related pages. HMO/web/index.html and HMO/web/landlords.html
Transaction The user needs to do something — apply, pay, report, register. HMO/web/apply.html
Information You are explaining something — policy, eligibility, standards, process. HMO/web/residents.html, licensing.html, standards.html, planning.html
FAQ You are answering the most common questions for a service area. HMO/web/faq.html

Tip: A user journey through a service typically starts at a service-landing page, branches into information pages, and ends at a transaction page. Map your user journeys before you start building so you know how many pages you need and which template each one uses.

Read the schema

A content schema is a JSON file with a hierarchical structure. Each branch of the tree represents a topic area; each leaf node represents a specific piece of content — typically a verbatim text string, a source reference and a document location.

Schema anatomy

A typical leaf node looks like this:

{
  "licensingRequirement": {
    "type": "object",
    "properties": {
      "verbatimText": {
        "type": "string",
        "const": "A licence is required for all HMOs that are
                  occupied by five or more people belonging to
                  2 or more households and who share a facility
                  or amenity."
      },
      "source": { "type": "string", "const": "SRC-02" },
      "documentLocation": {
        "type": "string",
        "const": "Section 2.0 – HMO Licensing"
      }
    }
  }
}

Three fields appear on nearly every node:

verbatimText
The exact approved wording. Use this text on the page — do not paraphrase unless the schema explicitly allows it.
source
A short reference code (e.g. SRC-02) that identifies which source document the content comes from.
documentLocation
The specific section or page within that source document so the content can be verified.

Navigating the schema tree

Large schemas are organised into top-level sections that mirror the content topics. For HMO the top-level sections are:

  • centralGovernmentLegislationAndGuidance — national law and guidance
  • councilStrategy — corporate plan commitments
  • localPolicy — local licensing policy, fire safety, space standards, planning
  • operationalGuidance — how the council processes applications
  • valueQualityConsistency — quality assurance framework

When building resident-facing pages, most of your content will come from localPolicy. When building landlord-facing pages you will also draw from centralGovernmentLegislationAndGuidance for legislative references.

Important: If a schema node is marked "status": "TODO", that content has not yet been captured. Do not fill in the gap with text you have written yourself — leave a visible placeholder or omit the section and note the gap in your implementation log.

Populate the template

Follow these steps once you have chosen a template and identified the relevant schema sections.

  1. 1

    Copy the template file

    Copy the template you need into your service folder and rename it. For example:

    cp templates/information.html HMO/web/licensing.html

    Never edit the original template files — they are the blank scaffolds that future services will also use.

  2. 2

    Fix the asset paths

    The templates link to assets using relative paths (assets/styles.css). If your page is in a sub-folder, update these paths to point back to the shared assets:

    <link rel="stylesheet" href="../../templates/assets/styles.css">
<link rel="stylesheet" href="../../templates/assets/a11y.css">
…
<script src="../../templates/assets/site.js"></script>
  3. 3

    Search for [PLACEHOLDER] and replace

    Every bracketed string in the template is a content placeholder. Use your editor's find-and-replace (Ctrl+H or Cmd+H) to find [PLACEHOLDER] as a pattern, then work through each one and replace it with text from the schema.

    Common placeholders in every template:

    • [Service name] — used in the breadcrumb, hero and footer. Example: Houses in Multiple Occupation (HMO)
    • [Topic title] — the page <h1>. Example: HMO licensing in Gloucester
    • [One sentence describing…] — the <meta name="description"> content
  4. 4

    Map schema sections to page sections

    Go through the template block by block and identify which schema path provides the content for each block. The information template has four section blocks by default — rename and populate each one with verbatim text from the schema.

    Example mapping for HMO/web/licensing.html:

    Template block

    Section 1 — general overview paragraph

    Schema path

    localPolicy.licensingPolicy.licensingRequirement.verbatimText

    Template block

    Key facts definition list — fees

    Schema path

    localPolicy.licensingPolicy.fees.feesFrom1April2026.default

    Template block

    Warning panel — licence conditions

    Schema path

    localPolicy.licensingPolicy.licenceConditionsStatement.verbatimText

  5. 5

    Delete blocks you do not need

    The templates include more building blocks than most pages will use. Once you have populated the blocks you need, delete any that are empty or not required. Common blocks that are sometimes removed: the data table, the numbered steps list, the CTA block at the bottom.

  6. 6

    Update the breadcrumb and navigation

    Update the breadcrumb so the final item matches the page's <h1> and carries aria-current="page". Update the sidebar "On this page" list to match the actual section headings you used.

Schema source metadata

Every content block on a page must be annotated with the schema path that provided its content. This creates a traceable link between the page and its authorised source, so editors and reviewers can verify content without reading HTML.

The two-part annotation pattern

Use both an HTML attribute and an HTML comment on every block:

<!-- schema-source: localPolicy.licensingPolicy.licensingRequirement | SRC-02 -->
<section id="do-you-need"
         data-schema-source="localPolicy.licensingPolicy.licensingRequirement">
  <h2>Do you need a licence?</h2>
  <p>A licence is required for all HMOs that are occupied by
     five or more people belonging to 2 or more households
     and who share a facility or amenity.</p>
</section>

The two parts serve different audiences:

HTML attribute data-schema-source
Machine-readable. Can be queried by scripts to audit page coverage, or by browser DevTools to inspect content traceability. The value is the dot-notation path to the schema node.
HTML comment <!-- schema-source: … -->
Human-readable in the source code. The comment includes both the schema path and the original source document reference code (e.g. SRC-02), making it easy for a developer editing the file to see where content came from without opening the schema.

Annotation granularity

Annotate at the block level — each <section>, <div> or <table> that contains schema-driven content should carry the attribute. You do not need to annotate individual <p> or <li> elements unless a single block draws from two different schema paths, in which case annotate the inner elements separately.

<!-- schema-source: localPolicy.licensingPolicy.preGrantConditions | SRC-02 -->
<section id="what-council-checks"
         data-schema-source="localPolicy.licensingPolicy.preGrantConditions">
  <h2>What the council checks</h2>
  <p>Before granting a licence the Council will determine…</p>

  <!-- schema-source: localPolicy.licensingPolicy.licenceApplicationGuidanceNotes
                      .namedPersonRequirement | SRC-05 -->
  <div data-schema-source="localPolicy.licensingPolicy
                           .licenceApplicationGuidanceNotes
                           .namedPersonRequirement">
    <div class="inset-panel info">
      <p><strong>Named person requirement:</strong>
         Gloucester City Council will only accept licence applications
         with a named person as the licence holder.</p>
    </div>
  </div>
</section>

What if content comes from no schema node?

Navigation elements, breadcrumbs, footers, and boilerplate UI text (button labels, skip links) do not originate from a content schema and need no annotation. Structural HTML added for layout purposes is also unannotated. If you find yourself writing content text that has no schema source, stop — either find the correct schema node or add the content to the schema first.

HMO worked example

The HMO (Houses in Multiple Occupation) service demonstrates this workflow end-to-end. All eight pages in HMO/web/ were built directly from HMO/hmo_master_schema_v1.json. No content was written by hand — every sentence, fee, standard and policy reference is traceable to a schema node.

The schema used

Schema file: HMO/hmo_master_schema_v1.json

This is the master schema — a 383 KB JSON file that aggregates five constituent sub-schemas (central government legislation, council strategy, local policy, operational guidance and value/quality/consistency). Each top-level section maps to a different layer of authoritative content.

The two user journeys

Before building any pages, the service team mapped two primary user journeys:

Resident journey

Users who already live in an HMO or are considering renting one. Their needs: understand what an HMO is, know their rights, find out if their property should be licensed, and report a problem.

Landlord journey

Property owners and managers who run or want to run an HMO. Their needs: find out if they need a licence, understand what standards apply, know the planning rules, and submit an application.

How the content was mapped — a step-by-step trace

The following example traces exactly how the fee table in licensing.html was built from the schema.

Step 1 — Find the schema node for fees.

Navigate to properties → localPolicy → properties → licensingPolicy → properties → fees → properties → feesFrom1April2026 in the schema. The node contains a default array with two rows:

"feesFrom1April2026": {
  "type": "array",
  "description": "Fees effective from 1 April 2026 as published on SRC-01",
  "default": [
    { "type": "New",     "duration": "up to 5 years", "fee": "£1260.00" },
    { "type": "Renewal", "duration": "up to 5 years", "fee": "£900.00"  }
  ],
  "source": "SRC-01",
  "documentLocation": "Section: HMO Licensing Fees – table effective from 1 April 2026"
}

Step 2 — Choose the right template block. Tabular data maps to the <table class="gov-table"> block in information.html. The table needs a caption (from the schema's description field), column headers (from the default array's property names) and rows (one per array entry).

Step 3 — Write the HTML and annotate it.

<!-- schema-source: localPolicy.licensingPolicy.fees | SRC-01 -->
<section id="fees" data-schema-source="localPolicy.licensingPolicy.fees">
  <h2>Fees</h2>
  <table class="gov-table">
    <caption>HMO licence fees from 1 April 2026</caption>
    <thead>
      <tr>
        <th scope="col">Licence type</th>
        <th scope="col">Duration</th>
        <th scope="col" class="num">Fee</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <th scope="row">New licence</th>
        <td>Up to 5 years</td>
        <td class="num">£1,260.00</td>
      </tr>
      <tr>
        <th scope="row">Renewal</th>
        <td>Up to 5 years</td>
        <td class="num">£900.00</td>
      </tr>
    </tbody>
  </table>
</section>

This pattern — find schema node → identify template block → write HTML → add annotation — is repeated for every section of every page.

Resident vs landlord: splitting the same schema into two journeys

The HMO schema contains information relevant to both residents and landlords. The key design decision was: do not show landlord information to residents and vice versa. Content was sorted into three buckets:

  • Resident-only — rights as a tenant, how to report problems, what your landlord must do. Drawn from localPolicy.tenantEnforcementContact and localPolicy.licensingPolicy.tenantLicensingStatement.
  • Landlord-only — how to apply, fees, fit-and-proper person test, planning requirements. Drawn from localPolicy.licensingPolicy, localPolicy.planningPolicy and operationalGuidance.applicationReceiptAndTriage.
  • Shared — what an HMO is, safety standards, FAQ. These appear on pages accessible from both journeys.

The service landing page (index.html) presents two clear entry points — "I'm a resident" and "I'm a landlord" — so the user self-selects into the right journey from the start.

Checklist before you publish

Run through this list before merging or publishing any schema-driven page.

Content

  • Every piece of body text traces back to a verbatimText or default value in the schema.
  • No content has been written that does not appear in the schema — no paraphrasing, no invented examples, no bridging sentences drawn from general knowledge.
  • Any TODO nodes in the schema are either omitted from the page or flagged with a visible placeholder (e.g. an inset panel saying "this section will be updated when the policy is confirmed").
  • Contact details (phone, email, opening hours) match the values in localPolicy.tenantEnforcementContact exactly.
  • Fees match the correct year's fee table in the schema — check which feesFrom… node is current.

Schema source annotations

  • Every content block has a data-schema-source attribute with a dot-notation path to the schema node.
  • Every content block has a corresponding <!-- schema-source: … | SOURCE-REF --> HTML comment above it.
  • Navigation, footer, breadcrumb and UI chrome have no data-schema-source annotation (correct — they are structural, not content).

Template requirements

  • No [PLACEHOLDER] strings remain anywhere in the file.
  • The <title> is under 60 characters and follows the pattern Page title — Gloucester City Council.
  • The <meta name="description"> is under 160 characters.
  • The breadcrumb's last item carries aria-current="page" and matches the <h1>.
  • Asset paths point to ../../templates/assets/ (or the correct relative path for the file's location).

Accessibility

  • Tab through the page — every interactive element shows a visible focus ring.
  • There is exactly one <h1> on the page and headings do not skip levels.
  • Every <table> has a <caption> and all header cells use scope="col" or scope="row".
  • All images have meaningful alt text; decorative images use alt="".
  • The page is readable at 320 px wide with no horizontal scrolling.
  • Run an automated check using axe DevTools, Lighthouse or WAVE.
